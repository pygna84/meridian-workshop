1. Overview

The inventory dashboard is a single-page web application backed by a Python API. Both halves live in one Git repository and run as separate processes during development. There is no database today — the API loads data from JSON files on startup and serves it from memory. This is suitable for the engagement and known operational scale; persistence is flagged as recommended future work in §7.

Frontend

Vue 3 · Vite · Composition API

  • Port 3000 in development
  • 8 view components, 10 reusable components
  • 4 composables for cross-cutting state
  • en + ja locale bundles

Backend

FastAPI · Pydantic · Uvicorn

  • Port 8001 in development
  • 16 REST endpoints across 7 domains
  • Pydantic models on every response
  • Auto-generated OpenAPI at /docs

Data layer

In-memory · loaded from JSON

  • 7 JSON files in server/data/
  • Loaded by mock_data.py at startup
  • No persistence between restarts
  • Purchase orders are a writable in-memory list

Testing

Pytest (backend) · Playwright (e2e)

  • tests/backend/ — API contract tests
  • tests/e2e/ — 20 browser tests, ~33s
  • Serialized: shared in-memory state
  • Run locally or with E2E_AUTOSTART=1 in CI

3. Frontend layout

The Vue 3 application uses the Composition API throughout. Cross-cutting state lives in composables; views consume composables and the centralized API client. UI strings flow through useI18n; design tokens (light + dark) live in CSS variables in App.vue and ripple through every component automatically.

4. Backend surface

FastAPI exposes 16 endpoints across 7 domains. Filtering on warehouse, category, status, and month is shared via the apply_filters / filter_by_month helpers; reports use a separate _filter_orders helper added during R1 remediation. Pydantic models on response enforce the contract for the frontend and the auto-generated OpenAPI doc at http://localhost:8001/docs.

5. Key data flows

Filtering across views

The FilterBar writes to four refs in the singleton useFilters composable. Views import the composable and watch the same refs — any filter change triggers a reload through api.js, which appends the active filters as query params. The backend resolves them through the shared filter helpers and returns a filtered payload. Filter state persists across navigation by virtue of the singleton.

Restocking recommendations

The /api/restocking/recommendations endpoint inspects each inventory item, computes a target par stock (round(reorder_point × 1.5)), adds a forecast lift when the SKU trends upward, sorts by deepest shortfall, and walks the candidate list against the operator-supplied budget. Items that fit go into recommendations; the rest into deferred. The frontend renders both lists; clicking Create PO opens a modal that posts to /api/restocking/purchase-orders and appends to the in-memory purchase_orders list.

Internationalisation and theming

Every UI string flows through useI18n; locales share a strict shape so missing keys can be detected. Currency formatting is locale-aware (USD ↔ JPY) using Intl.NumberFormat. The theme toggle in the header writes data-theme="dark|light" on the root and persists to localStorage; every component picks up the new palette automatically because all colors flow through CSS variables defined in App.vue.

7. Recommended next steps

Database migration. JSON-file persistence is fine for the engagement and the current scale (32 SKUs, ~250 orders), but it caps at single-process and forgets writes on restart. We recommend Postgres with SQLAlchemy as a follow-up; the in-memory filter_* helpers map cleanly to indexed queries.
  • CI pipeline. The Playwright suite runs in ~33s and is CI-ready (E2E_AUTOSTART=1). Wiring it into GitHub Actions converts IT's gatekeeper posture into a status-check posture.
  • Auth. The mock useAuth needs to be replaced before any production exposure. Recommend OIDC against Meridian's existing identity provider.
  • Theming sweep on view-level CSS. Restocking, Reports, and the PO modal carry scoped blues that the global brand variables don't reach. A single pass to migrate scoped CSS to the same tokens would complete D1.
  • Sticky “PO created” state. Today the badge lives in component memory. Adding GET /api/restocking/purchase-orders?source=restocking would let the Restocking view restore state on reload.
  • Backlog page wiring. The view exists and is now fully translated, but no nav link or route. Trivial to expose if Operations wants it.
  • Visual regression on the Reports bar chart. The structural test exists but a percy/Playwright screenshot diff would catch styling regressions earlier.
